# Архитектура, модули, FSD

## 1. Проблемы проектов без архитектуры

```jsx
// Без архитектуры (всё в одной куче):
src/
    components/
        Header.jsx
        Button.jsx
        UserCard.jsx
        ProductCard.jsx
        CartIcon.jsx
        LoginForm.jsx
        ProductList.jsx
        UserProfile.jsx
        // ... (50+ компонентов в одной папке)
    utils/
        helpers.js (1000+ строк разных функций)
    App.jsx

// ========== Проблемы ==========
// 1. Сложно масштабировать код
//    - Компонентов становится больше
//    - Логика дублируется
//    - Любое изменение ломает другие части проекта

// 2. Непонятная иерархия
//    - Неясно, где API, бизнес-логика, UI, состояние
//    - Разработчик тратит время на поиск файлов вместо работы

// 3. Долгое погружение в проект (onboarding занимает дни/недели)

// 4. Кольцевые зависимости
```

## 2. Кольцевые зависимости (Circular Dependencies)

```jsx
// Кольцевая зависимость — когда модули зависят друг от друга по кругу

// ========== Пример ==========
// user.js
import { getPosts } from "./post";

export function getUser(id) {
  /* ... */
}

// post.js
import { getUser } from "./user"; // ← обратный импорт!

export function getPosts(userId) {
  /* ... */
}

// Получается цикл: user → post → user

// ========== Чем опасны ==========
// 1. Ошибки инициализации — некоторые значения становятся undefined
// 2. Сложно тестировать — модули сильно связаны
// 3. Ломается переиспользуемость — нельзя вынести модуль отдельно
// 4. Сложно понимать зависимости
// 5. Скрытые баги

// =========! Главное правило =========!
// Верхний слой может зависеть от нижнего, но НЕ наоборот
// Нижний слой НИКОГДА не импортирует верхний

// Пример (FSD):
// pages → widgets → features → entities → shared
// ✅ можно
// ❌ нельзя: entities → features
```

## 3. Public API модуля

```jsx
// Public API — официальный набор того, что модуль разрешает использовать снаружи
// Обычно это файл index.js / index.ts

// ========== Без Public API ==========
// features/auth/ui/LoginForm.jsx
// features/auth/model/useAuth.js

// Импорт:
import { LoginForm } from '@/features/auth/ui/LoginForm';
import { useAuth } from '@/features/auth/model/useAuth';

// ❌ Проблема: если папка изменится — все импорты сломаются

// ========== С Public API ==========
// features/auth/index.js
export { LoginForm } from './ui/LoginForm';
export { useAuth } from './model/useAuth';
export type { User } from './types';

// Теперь импорт:
import { LoginForm, useAuth } from '@/features/auth';

// ========== Зачем нужен Public API ==========
// 1. Скрывает внутреннюю структуру — можно менять папки внутри модуля
// 2. Контролирует доступ — видно, что можно использовать, а что приватное
// 3. Упрощает импорты:
//    было: import { Button } from '../../../shared/ui/Button/Button';
//    стало: import { Button } from '@/shared/ui';
// 4. Помогает избегать циклических зависимостей
```

## 4. Иерархия компонентов

```jsx
// Главная идея — делить проект не по типам файлов, а по бизнес-смыслу

// ========== Плохой вариант ==========
// components/
// hooks/
// api/
// types/
// utils/

// При росте проекта всё смешивается

// ========== Лучше — Feature/Module подход ==========
// features/
// ├── auth/
// │   ├── api/
// │   ├── model/
// │   ├── ui/
// │   ├── types/
// │   └── index.js
// ├── profile/
// ├── comments/
// └── notifications/

// Каждая фича содержит всё необходимое внутри себя
```

## 5. Модульная архитектура

```jsx
// Модульная архитектура — подход, где приложение делится на независимые модули

// ========== Пример ==========
// modules/
// ├── auth/        (авторизация)
// ├── users/       (пользователи)
// ├── products/    (товары)
// └── orders/      (заказы)

// ========== Плюсы ==========
// ✅ Проще поддержка
// ✅ Независимость модулей
// ✅ Переиспользование
// ✅ Легче масштабировать

// ========== Минусы ==========
// ❌ Нет строгих правил зависимостей
// ❌ Со временем модули могут начать импортировать друг друга хаотично
```

## 6. FSD (Feature-Sliced Design)

```jsx
// FSD — архитектура для frontend-приложений со строгими слоями и правилами

// ========== Слои FSD ==========
src/
├── app/           # Глобальные настройки (роутинг, провайдеры, store, тема)
├── pages/         # Страницы приложения
├── widgets/       # Крупные UI-блоки (Sidebar, Header, Dashboard)
├── features/      # Пользовательские действия (login, add-to-cart, create-post)
├── entities/      # Бизнес-сущности (User, Product, Post)
└── shared/        # Переиспользуемые вещи (UI-kit, helpers, hooks, config)

// ========== Главное правило FSD ==========
// Слой может импортировать ТОЛЬКО нижележащие слои

// ✅ Можно: pages → widgets → features → entities → shared
// ❌ Нельзя: entities → features
// ❌ Нельзя: features → widgets

// ========== Когда использовать FSD ==========
// ✅ Средние и большие проекты
// ✅ Командная разработка
// ✅ Сложная бизнес-логика
// ✅ Долгоживущие проекты

// ========== Когда FSD избыточен ==========
// ❌ Pet-project
// ❌ Маленький лендинг
// ❌ Тестовое задание
// (тогда проще использовать модульную структуру)
```

---

## 7. Сравнение подходов

| Подход                    | Плюсы                                 | Минусы                     |
| ------------------------- | ------------------------------------- | -------------------------- |
| **Простая структура**     | Быстро стартовать                     | Быстро превращается в хаос |
| **Модульная архитектура** | Удобно делить по фичам                | Нет строгих правил         |
| **Feature-Sliced Design** | Масштабируемость, строгие зависимости | Нужно время на изучение    |

## Шпаргалка

### Правила зависимостей

```
FSD:    pages → widgets → features → entities → shared
        (только сверху вниз, никогда снизу вверх)

Модульная:  (нет строгих правил, нужно контролировать руками)
```

### Public API

```jsx
// ✅ Хорошо
import { LoginForm } from "@/features/auth";

// ❌ Плохо
import { LoginForm } from "@/features/auth/ui/LoginForm";
```

### Признаки проблемной архитектуры

| Проблема              | Признак                           |
| --------------------- | --------------------------------- |
| Сложно найти файл     | 50+ компонентов в одной папке     |
| Кольцевые зависимости | A → B и B → A                     |
| Нет Public API        | Импорты через "../../../"         |
| Непонятная иерархия   | UI, API, хуки, типы в одном месте |
