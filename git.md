# 1. Что такое Git Flow?

**Git Flow** — это модель ветвления в Git, которая задаёт структуру работы с кодом в команде.

Она вводит **строгие типы веток**, чтобы разделить:

- разработку
- релизы
- хотфиксы

## Основные ветки:

### 1. `main` (или `master`)

- содержит **только стабильный код**
- это то, что уже в проде

### 2. `develop`

- основная ветка разработки
- сюда вливаются все фичи

---

## Вспомогательные ветки:

### 3. feature/\*

Используются для новых задач:

```
feature/login-page
feature/payment-integration
```

👉 создаются от `develop`

---

### 4. release/\*

Подготовка к релизу:

```
release/1.0.0
```

👉 здесь:

- фиксят баги
- не добавляют новые фичи

---

### 5. hotfix/\*

Срочные исправления в проде:

```
hotfix/critical-bug
```

👉 создаются от `main`

---

## Как выглядит поток:

```
feature → develop → release → main
                 ↘ hotfix → main
```

---

## Зачем это нужно:

- порядок в команде
- разделение ответственности
- безопасные релизы
- параллельная разработка

---

# 2. Что такое git fetch?

`git fetch` — это команда, которая **загружает изменения с удалённого репозитория, но НЕ применяет их**.

---

## Что делает fetch:

- скачивает новые коммиты
- обновляет remote-tracking ветки (`origin/main`)
- НЕ меняет твою текущую ветку

---

## Пример:

```
git fetch origin
```

После этого:

- `origin/main` обновился
- твой `main` — нет

---

## Важно:

👉 fetch = “посмотреть, что изменилось”
👉 merge/pull = “влить изменения”

---

## Разница:

| команда | что делает               |
| ------- | ------------------------ |
| fetch   | только скачивает         |
| pull    | скачивает + сразу мёржит |

---

## Когда использовать:

- хочешь проверить обновления без риска
- перед rebase/merge
- в сложных конфликтах

---

# 3. Что такое git rebase?

`git rebase` — это способ **перенести коммиты одной ветки поверх другой**.

---

## Простая идея:

Ты как бы говоришь:

> “Возьми мои изменения и поставь их на самую свежую версию другой ветки”

---

## Пример:

Ты работаешь в feature:

```
A --- B --- C (main)
         \
          D --- E (feature)
```

Потом main обновился:

```
A --- B --- C --- F --- G (main)
         \
          D --- E (feature)
```

---

## После rebase feature на main:

```
A --- B --- C --- F --- G
                         \
                          D' --- E'
```

👉 коммиты D и E переписываются (новые hash!)

---

## Отличие от merge:

| rebase               | merge             |
| -------------------- | ----------------- |
| переписывает историю | сохраняет историю |
| линейная история     | граф с ветвлением |
| чище история         | безопаснее        |

---

## Когда использовать:

- локальная ветка
- перед PR
- для чистой истории

---

## Когда НЕ использовать:

- в shared ветках (main, develop)
- если уже есть другие разработчики

---

# 4. Что делает git cherry-pick <hash-коммита>?

`git cherry-pick` — это команда, которая **берёт один конкретный коммит и применяет его в текущую ветку**.

---

## Идея:

> “Мне не нужна вся ветка — мне нужен только этот один коммит”

---

## Пример:

```
A --- B --- C --- D (main)
          \
           E --- F (feature)
```

Ты хочешь взять только F:

```
git cherry-pick F
```

---

## Результат:

```
A --- B --- C --- D
          \
           E --- F
                \
                 F' (в другой ветке)
```

👉 создаётся новый коммит F' с новым hash

---

## Что важно:

- копируется только **изменение (diff)** коммита
- создаётся **новый коммит**
- история не переносится целиком

---

## Когда используют:

- срочно взять багфикс из другой ветки
- перенести конкретный коммит в release
- точечное применение изменений

---

## Риски:

- легко запутать историю
- можно получить дубли изменений
- конфликтует при повторном merge

---

# Мини-сравнение всех команд

| Команда     | Что делает            |
| ----------- | --------------------- |
| fetch       | скачивает изменения   |
| rebase      | перестраивает историю |
| cherry-pick | берёт один коммит     |
| merge       | объединяет ветки      |

---
