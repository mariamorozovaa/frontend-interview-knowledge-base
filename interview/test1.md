## 1. React: мемоизация и оптимизация

### React.memo, useMemo, useCallback

**Что ты написала:**

> _Memo (если пропсы не меняются не вызывает ререндер), useMemo, useCallback (useMemo, useCallback есть массив зависимостей) - кеширует функцию, чтобы не было перерендера при каждом создании (функция в зависимости useeffect)_

**Расшифровка:**

| Инструмент      | Что делает                                               | Когда использовать                                                                                                    |
| --------------- | -------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **React.memo**  | Компонент перерисовывается только если пропсы изменились | Для компонентов, которые рендерятся часто и получают одни и те же пропсы. Оборачиваем **дочерний** компонент.         |
| **useMemo**     | Кеширует результат вычислений                            | Тяжёлые расчёты, фильтрация массивов, создание объектов/массивов. Пересчитывается только при изменении зависимостей.  |
| **useCallback** | Кеширует саму функцию (ссылку на функцию)                | Когда функция передаётся в дочерний компонент, обёрнутый в `React.memo`, или используется в зависимостях `useEffect`. |

**Пример из твоей легенды:**

```tsx
// Фильтрация большого списка — используем useMemo
const filteredRequests = useMemo(() => {
  return requests.filter((req) => req.status === "pending");
}, [requests]);

// Функция, которую передаём в дочерний компонент — useCallback
const handleApprove = useCallback((id: string) => {
  approveRequest(id);
}, []);
```

**Важно:** `useCallback` не предотвращает перерисовку сам по себе. Он нужен, чтобы ссылка на функцию не менялась при каждом рендере — тогда `React.memo` в дочернем компоненте сработает корректно.

---

> _"Child оборачивается в memo, Parent - в useMemo"_

**Правильно так:** Дочерний компонент оборачиваем в `React.memo`, а в родительском используем `useMemo` и `useCallback` для пропсов, которые передаём в ребёнка.

---

> _"useMemo - чтобы кешировать результат тяжелых вычислений, (при постоянном изменении пропсов бессмысленно) сортировка массивов, фильтрация списков, тяжелые вычисления, создание объектов/массивов, для стабильных пропсов"_

**Дополнение:** `useMemo` имеет смысл, если вычисления **действительно тяжёлые** (сотни/тысячи элементов). Для простых операций — оверхед.

### useContext vs Redux

**Что ты написала:**

> _useContext (против props drilling) но при изменении контекста перерендеривает все компоненты подписанные на контекст, useSelector (передача state), Reducer (менеджер состояний) - логика state, dispatch (type) - функция reducer, action: type и payload._

**Расшифровка:**

| Инструмент         | Особенность                                                                                                                                                                                 |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **useContext**     | Простой способ передать данные глубоко в дерево. Минус: при любом изменении контекста перерендериваются **все** подписанные компоненты.                                                     |
| **useReducer**     | Альтернатива `useState` для сложной логики. Состояние меняется через `dispatch(action)`. `action` — объект с `type` и `payload`.                                                            |
| **Redux Toolkit**  | Глобальное хранилище. `useSelector` — достать данные, `useDispatch` — отправить action. Не перерендеривает всё подряд, только те компоненты, которые используют изменённую часть состояния. |
| **Zustand / MobX** | Альтернативы Redux. Zustand — проще и легче, MobX — реактивный, менее шаблонный.                                                                                                            |

**Пример из твоей легенды (АРИДА):**

```tsx
// useReducer + Context
const [state, dispatch] = useReducer(reducer, initialState);

const Provider = ({ children }) => <Context.Provider value={{ state, dispatch }}>{children}</Context.Provider>;

// Где-то в компоненте
dispatch({ type: "SET_USER", payload: user });
```

```tsx
// Redux Toolkit (RTK) — твой стек
const user = useSelector((state) => state.user);
const dispatch = useDispatch();
dispatch(setUser(userData));
```

### useRef

**Что ты написала:**

> _useRef - за ним не следит реакт, не вызывает изменение состояния, но сохраняет значение в поле current (хранит ссылку на узел дом дерева) <div ref={refCount}> B refCount будет этот див_

**Расшифровка:**

- `useRef` хранит значение между рендерами, но **не вызывает перерендер** при изменении.
- Используется для:
  - Ссылка на DOM-элемент (`<div ref={myRef} />`)
  - Хранение таймеров (`timerRef.current = setTimeout(...)`)
  - Хранение предыдущего значения (`prevValueRef.current = value`)

```tsx
// Пример из твоей легенды (фильтрация с debounce)
const timerRef = useRef<NodeJS.Timeout | null>(null);

const handleSearch = (value: string) => {
  if (timerRef.current) clearTimeout(timerRef.current);
  timerRef.current = setTimeout(() => {
    fetchSearchResults(value);
  }, 500);
};
```

### Ключи в списках (key)

**Что ты написала:**

> _Отличие индекса и идентификатора: если index используем, то при удалении одного элемента, изменятся все индексы элементов дальше по списку, и будут перерендериваться все элементы списка, потому что они все по порядку, а если сделать айдишник, то он уникальный и перерендер всех элементов происходить не будет_

**Расшифровка:** ✅ Абсолютно верно! React использует `key`, чтобы идентифицировать элементы. Если использовать `index`, при удалении/добавлении элементов React путается и перерендеривает всё. Лучше использовать уникальный `id`.

```tsx
// ❌ Плохо
{
  items.map((item, index) => <Item key={index} {...item} />);
}

// ✅ Хорошо
{
  items.map((item) => <Item key={item.id} {...item} />);
}
```

## 2. Архитектура FSD (Feature-Sliced Design)

**Что ты написала:**

> _FSD самая актуальная, Shared > features > widget > pages_

**Расшифровка:**

**FSD (Feature-Sliced Design)** — методология организации кода в React-проектах.

**Слои (от низшего к высшему):**

| Слой         | Что содержит                                              | Пример                                  |
| ------------ | --------------------------------------------------------- | --------------------------------------- |
| **shared**   | Переиспользуемые утилиты, UI-компоненты, хуки, API-клиент | `Button`, `Input`, `useDebounce`, `api` |
| **entities** | Бизнес-сущности (модели данных)                           | `User`, `Request`, `Document`           |
| **features** | Пользовательские сценарии (действия)                      | `CreateRequest`, `ApproveDocument`      |
| **widgets**  | Блоки интерфейса (комбинации фич)                         | `RequestList`, `FileUploader`           |
| **pages**    | Страницы (комбинация виджетов)                            | `DashboardPage`, `RequestsPage`         |

**Правила:**

- Компонент высшего слоя НЕ может импортировать из низшего.
- `pages` → можно импортировать из `widgets`, `features`, `entities`, `shared`.
- `shared` → НЕ может импортировать из `widgets`, `features`, `pages`.

> _"Нельзя компонент высшего порядка импортировать в компонент низшего порядка"_

**Расшифровка:** ✅ Верно! Только сверху вниз: `pages → widgets → features → entities → shared`.

> _"index.tsx для экспорта (public API)"_

**Расшифровка:** В каждом слое есть `index.ts`, который экспортирует только то, что разрешено использовать другим слоям (public API). Скрытые детали реализации остаются внутри.

**Пример из твоей легенды (АРИДА):**

```
src/
├── shared/
│   ├── ui/ (AppButton, AppInput)
│   ├── api/ (RTK Query)
│   └── hooks/ (useDebounce)
├── entities/
│   ├── user/ (User type, useUser)
│   └── request/ (Request type, useRequest)
├── features/
│   ├── create-request/ (CreateRequestForm)
│   └── approve-request/ (ApproveButton)
├── widgets/
│   ├── request-list/ (RequestList)
│   └── file-uploader/ (FileUploader)
└── pages/
    ├── dashboard/
    └── requests/
```

## 3. Сети, HTTP, что происходит в браузере

### Что происходит при вводе URL в браузер

**Что ты написала:**

> _Tcp/udp (html) Dns рукопожатия_

**Расшифровка:** Полный путь:

1. **DNS-запрос** — браузер узнаёт IP-адрес по домену.
2. **TCP-рукопожатие (3 шага)** — установка соединения с сервером.
3. **TLS-рукопожатие** — если HTTPS, шифрование.
4. **HTTP-запрос** — браузер отправляет GET-запрос.
5. **Сервер обрабатывает** — генерирует HTML.
6. **Браузер получает ответ** — парсит HTML, строит DOM, CSSOM.
7. **Render Tree** — объединение DOM + CSSOM.
8. **Layout** — вычисление позиций элементов.
9. **Paint** — отрисовка пикселей.
10. **Composite** — объединение слоёв.

## 5. Оптимизация рендеринга: Reflow, Repaint, Composite

**Что ты написала:**

> _dom layout paint reflow, transform в отдельном слое_

**Расшифровка:**

| Этап                | Что происходит                | Как оптимизировать                                                                        |
| ------------------- | ----------------------------- | ----------------------------------------------------------------------------------------- |
| **DOM + CSSOM**     | Построение деревьев           | Минимум вложенности                                                                       |
| **Layout / Reflow** | Вычисление позиций и размеров | Меньше обращений к `offsetHeight`, `getBoundingClientRect()`                              |

> _"transform в отдельном слое"_ — ✅ Верно! Анимации через `transform: translate()` или `opacity` выполняются в GPU, не вызывая reflow/paint.

const obj = Object.create(null); // ✅ Нет __proto__, никаких унаследованных свойств

## 7. React: useReducer, Context, Redux

**Что ты написала:**

> _useReducer для доступа к состояниям из любого компонента_

**Уточнение:** `useReducer` сам по себе **не даёт доступ из любого компонента**. Его нужно обернуть в `Context`, чтобы передавать состояние и `dispatch` вниз по дереву.

```tsx
// useReducer + Context = доступ из любого компонента
const AppContext = createContext(null);

const AppProvider = ({ children }) => {
  const [state, dispatch] = useReducer(reducer, initialState);
  return <AppContext.Provider value={{ state, dispatch }}>{children}</AppContext.Provider>;
};
```

## 8. useCallback: когда нужен

**Что ты написала:**

> _useCallback: 1) memo дочерний компонент 2) Передается куда-то в качестве зависимостей_

**Расшифровка:** ✅ Верно!

1. **Дочерний компонент обёрнут в `React.memo`** — иначе смысла нет.
2. **Функция используется в зависимостях `useEffect`** — чтобы избежать бесконечного цикла.

```tsx
const fetchData = useCallback(async () => {
  const result = await api.getData();
  setData(result);
}, []);

useEffect(() => {
  fetchData();
}, [fetchData]); // зависимость — стабильная ссылка
```

## 10. Чистые функции, функции высшего порядка, каррирование

**Что ты написала:**

> _Чистая функция- при одних и тех же значениях один и тот же результат_

**✅ Верно.** Чистая функция — всегда возвращает одинаковый результат для одинаковых аргументов и не имеет побочных эффектов.

> _"Функция высшего порядка принимает и возвращает функцию"_

**✅ Верно.** Пример: `map`, `filter`, `reduce`, `useCallback`.

> _"Каррирование"_

**Каррирование** — преобразование функции, которая принимает несколько аргументов, в цепочку функций, каждая из которых принимает один аргумент.

```ts
// Обычная функция
const add = (a: number, b: number) => a + b;

// Каррированная
const curriedAdd = (a: number) => (b: number) => a + b;
curriedAdd(2)(3); // 5
```

## 12. Event Loop, микро- и макрозадачи

**Что ты написала:**

> _Event loop makro и микрозадачи_

**Расшифровка:**

- **Макрозадачи** (Macrotasks): `setTimeout`, `setInterval`, события (клики, скролл), запросы.
- **Микрозадачи** (Microtasks): `Promise.then`, `queueMicrotask`, `MutationObserver`.

**Порядок выполнения:**

1. Выполнить синхронный код.
2. Выполнить ВСЕ микрозадачи.
3. Взять ОДНУ макрозадачу.
4. Снова все микрозадачи → макрозадача → и т.д.

```js
console.log("1");
setTimeout(() => console.log("2"), 0);
Promise.resolve().then(() => console.log("3"));
console.log("4");
// 1 → 4 → 3 → 2
```

## 17. TypeScript: дженерики, утилитарные типы

**Что ты написала:**

> _Кастомные generic pick omit и тд, Как работают | и &, Interface и type, Extends implements, Generic - создать переиспользуемые типы_

**Расшифровка:**

### Базовые типы

| Тип                  | Описание                                                                          |
| -------------------- | --------------------------------------------------------------------------------- |
| `any`                | Отключает проверку типов (❌ не рекомендуется)                                    |
| `unknown`            | Безопасная альтернатива `any` — требует проверки перед использованием             |
| `never`              | Тип, который никогда не встречается (например, функция всегда выбрасывает ошибку) |
| `void`               | Функция ничего не возвращает                                                      |
| `null` / `undefined` | Отсутствие значения                                                               |

### Union и Intersection

```ts
type Status = "pending" | "approved" | "rejected"; // | (OR) — одно из
type Admin = { role: "admin" } & { permissions: string[] }; // & (AND) — всё сразу
```

### Interface vs Type

|                          | `interface`                 | `type` |
| ------------------------ | --------------------------- | ------ |
| Расширение               | `extends`                   | `&`    |
| Объединение (union)      | ❌ Нет                      | ✅ Да  |
| Работа с объектами       | ✅ Да                       | ✅ Да  |
| Может быть переопределён | ✅ Да (declaration merging) | ❌ Нет |

### Generic (Дженерики)

```ts
// Универсальная функция
function getValue<T>(value: T): T {
  return value;
}

const str = getValue<string>("hello"); // 'hello'
const num = getValue(42); // автоматический вывод
```

### Утилитарные типы

```ts
interface User {
  id: number;
  name: string;
  email: string;
}

type UserPreview = Pick<User, "id" | "name">; // только id и name
type UserWithoutEmail = Omit<User, "email">; // всё кроме email
type PartialUser = Partial<User>; // все поля необязательные
type RequiredUser = Required<User>; // все поля обязательные
type UserId = Pick<User, "id">; // { id: number }
```
