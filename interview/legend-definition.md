````markdown
## 🖥 Фронтенд (Интерфейс пользователя)

### React

Библиотека для сборки интерфейса. Разбивает страницу на мелкие переиспользуемые кусочки — компоненты (кнопки, формы, карточки). Позволяет управлять состоянием приложения и эффективно обновлять интерфейс при изменении данных.

## 🔌 Способ общения (Мост между фронтендом и бэкендом)

### REST API

Набор правил и стандартов, по которым фронтенд общается с бэкендом. Клиент отправляет HTTP-запросы (GET, POST, PUT, DELETE) на конкретные адреса (эндпоинты), например `POST /api/login`. Бэкенд обрабатывает запрос и возвращает ответ (обычно в формате JSON).

### RTK Query

Мощный инструмент для управления состоянием и кэширования данных на фронтенде. Входит в состав Redux Toolkit. Автоматически:

- генерирует хуки для запросов (`useGetUsersQuery`);
- кэширует ответы;
- инвалидирует кэш после мутаций (изменений);
- обрабатывает состояния загрузки и ошибок.

**Альтернатива:** TanStack Query (React Query).

## ⚙ Бэкенд (Серверная логика)

### NestJS

Фреймворк для создания серверных приложений на Node.js (TypeScript).

- Подходит для крупных Enterprise-проектов.

### Express

Минималистичный и самый популярный фреймворк для Node.js.

- Легкий, не навязывает структуру.

### .NET / ASP.NET Core (C#)

## 🗄 Хранение данных и Кэш

### PostgreSQL

Главная реляционная (табличная) база данных. Здесь надёжно хранятся критичные данные: профили пользователей, история заказов, балансы, связи между сущностями.

- Мощнее в плане сложных аналитических запросов.
- Лучше работает с нестандартными данными (например, JSON внутри таблиц).
- Строже следит за целостностью данных.

### MySQL

Одна из самых популярных СУБД в мире. Прямой конкурент PostgreSQL.

- Проще в настройке.
- Быстрее «из коробки» справляется с простыми операциями чтения.
- Огромное сообщество.

### Redis

Сверхбыстрая база данных в оперативной памяти. Используется для:

- **Кэширования** — чтобы не заставлять PostgreSQL заново искать часто запрашиваемые данные (например, список категорий).
- **Хранения временных данных** — сессий пользователей, кодов подтверждения.

## ✉ Асинхронные задачи (Очереди)

### RabbitMQ

Брокер сообщений (очередь). Используется для задач, которые не должны тормозить пользователя.
Пример: пользователь оформляет заказ — бэкенд мгновенно отвечает «Заказ принят!», а в фоне отправляет email, генерирует чек и уведомляет склад через RabbitMQ. Если сервер отправки писем упал — RabbitMQ сохранит задачу и выполнит её позже.

### Apache Kafka

Распределённая платформа потоковой передачи данных (лог сообщений).
**Главное отличие от RabbitMQ:** RabbitMQ создан для того, чтобы надёжно доставить сообщение и сразу его удалить. Kafka хранит все сообщения как бесконечную историю и позволяет читать их снова и снова.

## 📋 Управление задачами и процессами

### Jira

Система управления задачами (таск-трекер). Используется для:

- заведения задач;
- отслеживания статусов;
- планирования спринтов;
- ведения бэклога.

### Confluence

Вики-система для ведения документации. В больших компаниях аналитики описывают требования, бизнес-логику, API, макеты и архитектуру в Confluence, чтобы вся команда имела доступ к актуальной информации.

### Figma

Инструмент для создания макетов интерфейсов. Дизайнеры рисуют экраны, прототипы, UI Kit. Разработчики используют Figma как источник истины по внешнему виду и поведению элементов.

### Scrum

Одна из самых популярных гибких методологий управления разработкой. Основана на:

- **Спринтах** — фиксированных временных отрезках (обычно 1–2 недели).
- **Ежедневных встречах (Daily)**.
- **Планировании (Sprint Planning)**.
- **Ретроспективе (Retrospective)** — обсуждение, что пошло хорошо, а что можно улучшить.

### Спринт (Sprint)

Фиксированный период времени (обычно 1–4 недели), в течение которого команда выполняет запланированный объём задач.

### Backlog

Структурированный список всех задач, требований и доработок, которые стоят перед командой. Постоянно обновляется и приоритизируется.

### Grooming / Refinement

Регулярная встреча, на которой команда разбирает, детализирует и оценивает новые задачи из бэклога. Уточняют требования, обсуждают технические детали, оценивают сложность.

### Sprint Planning

Встреча в начале спринта, на которой команда:

- выбирает задачи из бэклога;
- оценивает их в Story Points;
- определяет цель спринта;
- формирует список задач для реализации.

### Retrospective

Встреча в конце спринта, на которой команда обсуждает:

- что пошло хорошо;
- что пошло плохо;
- что можно улучшить в следующем спринте.

## 🌿 Работа с Git

### Git Flow

Популярная методология работы с ветками в Git.

**Основные ветки:**

- `main` / `master` — стабильный код, готовый к релизу.
- `develop` — актуальная разработка, интеграция фич.

**Вспомогательные ветки:**

- `feature/название` — для новой функциональности.
- `fix/название` — для исправления багов.
- `hotfix/название` — для срочных исправлений в продакшене.
- `release/номер` — подготовка к релизу.

### Merge Request (MR) / Pull Request (PR)

Запрос на слияние изменений из одной ветки в другую. Процесс включает:

- создание MR/PR;
- Code Review коллегами;
- обсуждение и правки;
- апрув (одобрение);
- слияние.

### Code Review

Процесс проверки кода другими разработчиками. Цели:

- найти ошибки и баги;
- улучшить качество кода;
- поделиться знаниями;
- соблюдать стандарты и архитектуру.

### Jenkins

Инструмент для непрерывной интеграции (CI). Автоматизирует:

- сборку проекта;
- запуск тестов;
- деплой на стенды.
  Разработчик может собрать свою ветку на тестовом стенде через Jenkins.

## 🔐 Безопасность и доступы

### RBAC (Role-Based Access Control)

Модель управления доступом, основанная на ролях.

- У каждого пользователя есть одна или несколько ролей.
- У каждой роли есть набор прав (permissions).
- Интерфейс строится динамически на основании прав.

Пример:

- `Admin` → может управлять пользователями.
- `Manager` → может согласовывать заявки.
- `Employee` → только просмотр.

### Permissions (Права / Разрешения)

Конкретные действия, которые разрешены пользователю. Например:

- `users:view` — просмотр списка пользователей.
- `users:edit` — редактирование пользователей.
- `requests:approve` — согласование заявок.

## 🧪 Тестирование

### Jest

Фреймворк для написания юнит-тестов на JavaScript/TypeScript.

- Популярный стандарт в React-экосистеме.
- Позволяет писать тесты для отдельных функций и компонентов.
- Имеет встроенный ассерт (проверки) и поддержку моков (заглушек).

### React Testing Library

Библиотека для тестирования React-компонентов. Главный принцип — тестировать поведение, а не детали реализации. Например, проверять, что кнопка отображается, а не что внутри неё определённый CSS-класс.

### Unit-тесты (Юнит-тесты)

Тестирование отдельных модулей (функций, компонентов) изолированно от остальной системы.

### E2E-тесты (End-to-End)

Сквозные тесты, проверяющие работу всего приложения целиком (например, пользователь авторизуется → переходит на страницу → выполняет действие).

## 🧩 Формы и динамика (React Hook Form)

### React Hook Form (RHF)

Библиотека для управления формами в React/Next.js.

- Использует **неуправляемые** компоненты через `ref`, что делает формы быстрыми.
- Перерисовывается только то поле, в котором пишет пользователь, а не вся форма.
- Минимум шаблонного кода: `<input {...register("email")} />`.

### register

Метод из React Hook Form, который «регистрирует» поле формы. Возвращает пропсы для инпута (`onChange`, `onBlur`, `ref`). Пример: `<input {...register("username")} />`.

### Controller

Компонент-обёртка из React Hook Form, который позволяет подружить сторонние UI-библиотеки (Ant Design, Material UI) с логикой формы. Пример использования:

```jsx
<Controller name="status" control={control} render={({ field }) => <Select {...field} />} />
```
````

### FormProvider

Компонент-контейнер (провайдер контекста), который позволяет глубоко вложенным компонентам получать доступ к методам и данным формы без передачи через пропсы (props drilling).

### useFieldArray

Хук из React Hook Form для работы с динамическими массивами полей. Позволяет:

- добавлять (`append`);
- удалять (`remove`);
- менять порядок (`move`).
  Автоматически присваивает уникальный `id` каждому элементу.

### watch

Метод из React Hook Form, который отслеживает значения полей в реальном времени. Можно следить за всей формой или за конкретными полями. Перерисовывает **весь компонент формы**.

### useWatch

Хук-альтернатива `watch`, который работает на уровне отдельного дочернего компонента. Перерисовывает **только этот дочерний компонент**, а не всю форму. Используется для оптимизации производительности больших форм.

### Yup

Библиотека для описания схем валидации данных. Позволяет создавать правила валидации в декларативном стиле. Пример:

```js
const schema = Yup.object({
  email: Yup.string().email().required(),
  age: Yup.number().min(18).max(100),
});
```

**Статус:** проверенная классика, но постепенно уступает место Zod.

### Zod

Библиотека для валидации данных с использованием TypeScript. Современный стандарт, идеально подходит для проектов на TypeScript. Позволяет выводить типы автоматически:

```ts
const schema = z.object({
  email: z.string().email(),
  age: z.number().min(18).max(100),
});
type User = z.infer<typeof schema>;
```

## 📊 Оптимизация и UX

### Skeleton (Скелетон)

Элемент интерфейса, который показывает схематичный серый макет страницы во время загрузки данных.
**Преимущества:**

- резервирует место под контент — интерфейс не «прыгает»;
- создаёт ощущение, что сайт уже загрузился, просто контент «проявляется»;
- воспринимается быстрее и комфортнее, чем обычный спиннер (крутилка).

### React.memo

Функция высшего порядка (HOC), которая мемоизирует компонент. Если пропсы не изменились, React пропускает перерисовку этого компонента. Используется для оптимизации производительности, особенно в списках и карточках.

### useMemo

Хук, который мемоизирует результат вычислений. Вычисляет значение только тогда, когда меняются зависимости. Используется для дорогих вычислений (подготовка данных для графиков, фильтрация больших массивов).

### Debounce (Отложенный вызов)

Техника ограничения частоты вызовов. Например, при поиске: пользователь вводит текст, запрос на сервер отправляется только через 500 мс после остановки ввода. Это уменьшает количество запросов и нагрузку на сервер.

### Polling (PollingInterval)

Механизм периодического опроса сервера. Например, приложение каждые 30 секунд запрашивает новые уведомления. В RTK Query настраивается через `pollingInterval`. Это альтернатива WebSocket для не критичных по скорости обновлений.

## 👥 Роли в команде

### Product Owner

Роль внутри Scrum-команды. Фокусируется на тактике «здесь и сейчас»:

- ежедневно работает с разработчиками;
- детально расписывает задачи;
- следит за текущим прогрессом.
  **Часто совмещается с ролью Product Manager.**

### Product Manager (Продуктовод)

Мыслит глобально и стратегически:

- исследует рынок и конкурентов;
- считает экономику продукта;
- определяет долгосрочную стратегию (Roadmap);
- решает, куда продукт движется в разрезе месяцев и лет.

### Project Manager (Руководитель проекта)

Отвечает за сроки, бюджет и процессы. Вопросы:

- «Успеем ли мы к дедлайну?»
- «Хватает ли нам людей?»
- «Почему процесс застрял?»

**Коротко:**

- Product Manager → решает **что** делать и **зачем**.
- Product Owner → решает **как именно** и **в каком порядке**.
- Project Manager → организует процесс, чтобы это было сделано **вовремя**.

### Team Lead (Тимлид)

Руководитель команды разработчиков. Отвечает за:

- технические решения;
- Code Review;
- менторство;
- релизы;
- взаимодействие с другими командами;
- распределение задач.

### DevOps

Специалист по автоматизации инфраструктуры, сборки, тестирования и доставки кода. Настраивает CI/CD, Jenkins, Kubernetes, управляет окружениями.

### QA / AQA

**QA (Quality Assurance)** — специалист по ручному тестированию. Проверяет работу приложения, находит баги, оформляет баг-репорты.  
**AQA (Automated QA)** — пишет автотесты (UI-, API-, интеграционные), автоматизирует процесс проверки.

### System Analyst

Отвечает на вопрос: «Как это будет работать?» Продумывает:

- взаимодействие сервисов;
- API;
- структуру баз данных;
- архитектуру решения в целом.

### Business Analyst

Отвечает на вопрос: «Что именно хочет бизнес?» Собирает требования, описывает бизнес-процессы и сценарии использования.

### UI/UX Designer

**UI (User Interface)** — проектирует внешний вид интерфейса: цвета, шрифты, кнопки, отступы.  
**UX (User Experience)** — проектирует пользовательский опыт: логику перемещения по сайту, сценарии, удобство использования.

## 📐 Оценка задач

### Story Points (SP)

Относительная единица измерения сложности задачи в Agile-командах. Учитывается:

- объём работы;
- техническая сложность;
- риски;
- степень неопределённости.

**Примерная шкала (числа Фибоначчи):**

- 1 SP — небольшое изменение текста или простой баг.
- 3 SP — небольшая форма или простой экран.
- 5 SP — полноценный CRUD.
- 8 SP — сложная бизнес-логика или интеграция с несколькими API.
- 13 SP — очень большая задача, рекомендуется разбить на несколько.

### Planning Poker

Техника совместной оценки задач в Scrum. Процесс:

1. Аналитик рассказывает задачу.
2. Команда обсуждает и задаёт вопросы.
3. Каждый разработчик выбирает оценку (1, 2, 3, 5, 8, 13...).
4. Все одновременно показывают оценки.
5. Если оценки различаются — обсуждают причины.
6. Проводят повторное голосование до достижения консенсуса.

## 5. Вопросы-ловушки от хитрого тимлида (и как на них отвечать)

Тимлиды часто задают каверзные вопросы, чтобы проверить, **понимаешь** ли ты термины, а не просто заучила.

| Вопрос-ловушка                                                                                                  | Что на самом деле хотят услышать                                                                                                                                                               |
| --------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **"Почему вы используете Story Points, если это просто часы, замаскированные под числа?"**                      | Ты объясняешь, что Story Points — не часы. Один разработчик сделает за 4 часа, другой — за 8. SP — это сложность + риски + неопределённость. Это позволяет сравнивать задачи внутри команды.   |
| **"А почему у вас такая сложная система с RBAC? Проще же просто проверять роль и показывать/скрывать кнопки."** | Ты объясняешь, что у пользователя может быть несколько ролей, а права могут меняться. Проверка по ролям — жёсткая, а по permissions — гибкая. Назначать права проще, чем создавать новые роли. |
| **"Зачем вам Skeleton, если можно просто крутилку поставить?"**                                                 | Skeleton резервирует место → интерфейс не прыгает → пользователь видит структуру → воспринимается быстрее. Это доказано UX-исследованиями.                                                     |
| **"Вы используете RTK Query. А почему не просто fetch в useEffect?"**                                           | RTK Query автоматически кэширует, инвалидирует кеш, управляет состоянием загрузки, избегает дублирования запросов. Вручную это писать долго и ошибкоопасно.                                    |
| **"Зачем вам React Hook Form? Обычный useState же проще."**                                                     | useState перерисовывает всю форму на каждый символ. RHF через ref обновляет только нужное поле, особенно важно для больших форм.                                                               |
| **"В чём разница между watch и useWatch?"**                                                                     | `watch` перерисовывает весь компонент формы. `useWatch` — только тот дочерний компонент, где используется. Для больших форм критично.                                                          |
| **"А почему вы не используете WebSocket для всего? Это же круто."**                                             | WebSocket требует постоянного соединения, это дороже. Для данных, которые не требуют мгновенного обновления (например, список пользователей), достаточно polling или обычных запросов.         |
| **"Как вы тестируете формы? Просто проверяете, что они отправились?"**                                          | Тестируем валидацию, динамические поля, зависимости между полями, состояния ошибок, корректную отправку данных. Юнит-тесты + интеграционные.                                                   |
| **"Почему вы оцениваете задачи командой, а не тимлид сам?"**                                                    | Разные разработчики видят разные риски. Кто-то знает этот модуль, кто-то нет. Командная оценка точнее и учитывает больше точек зрения.                                                         |
| **"А если задача оценена в 8 SP, а разработчик сделал за 2 дня — это значит, что мы ошиблись?"**                | Нет. SP — это не часы. Возможно, разработчик знал этот модуль, возможно, задача оказалась проще. Важно смотреть на скорость команды (velocity) в среднем, а не на отдельные задачи.            |
